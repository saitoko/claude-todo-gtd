## 1. GTD ってなに？

### 「頭の中に置かない」という考え方

やるべきことが頭の中にある状態を想像してください。「あれもやらなきゃ」「これも忘れてた」という状態が続くと、今やっていることに集中できなくなります。

**GTD（Getting Things Done）** は、頭の中にあるものを全部外に出して、「決まった引き出し」に仕分けることで、脳の無駄な負荷をなくす考え方です。

引き出しに入れてしまえば、頭の中から追い出せます。必要なときに引き出しを開ければいいので、「忘れないようにしなければ」というエネルギーが不要になります。

### 引き出しの種類

`/todo` スキルは6つの引き出し（カテゴリ）を使います：

| ラベル | 日常語 | こんなときに使う |
|--------|--------|----------------|
| `inbox` | **とりあえずボックス** | まだ仕分けていないもの。「とにかく忘れないうちに記録したい」とき |
| `next` | **次にやること** | 「これが次の作業だ」と決めたもの。具体的なアクション |
| `waiting` | **返事待ちボックス** | 誰かに依頼した、または外部の完了を待っているもの |
| `someday` | **夢リスト** | 急がないけど忘れたくないこと。いつかやるかもしれないもの |
| `project` | **大きな案件フォルダ** | 複数の作業ステップが必要な案件・目標 |
| `reference` | **資料棚** | やることではなく、後で見返したい情報・メモ |

### コンテキスト（`@`ラベル）とは

コンテキストとは、「どこで・何があるときにできるか」を表す目印です。

たとえば *@PC* をつけておくと、パソコンが使えるときに「今できることは何だろう？」とフィルタするだけで、その場でできるタスクだけが表示されます。

- 外出中に電話できるタスクだけ見たい → `/todo list` *@外出中*
- 自宅でしかできないことを確認したい → `/todo list` *@自宅*

最初は3〜4個から始めるのがおすすめです。例：*@PC*、*@外出中*、*@自宅*

## 2. クイックスタート（5分で始める）

### ステップ1：頭の中のことを記録する

まず、今頭の中にある「やらなければならないこと」を全部 inbox に入れます。仕分けは後でいいので、とにかく記録することを優先してください。

- `/todo inbox` *上司にメールを返す*
- `/todo inbox` *来週の会議の準備*
- `/todo inbox` *旅行の計画を立てる*
- `/todo inbox` *読みかけの本を読み終える*

### ステップ2：リストを確認する

`/todo list`

inbox に入ったタスクが一覧表示されます。

### ステップ3：1つだけ「次にやること」に移す

「今週中に必ずやること」を1つ決めて、next に移します。Issue の番号（リスト表示の `#番号`）を指定します。

`/todo move` *1* `next`

これだけで OK です。あとは日々の作業の中で少しずつ使い方を覚えていきましょう。

```
頭の中 → /todo inbox → とりあえずボックス
                              ↓
                        /todo move → next（次にやること）
                                     waiting（返事待ち）
                                     someday（いつかやるかも）
                                          ↓
                                    /todo done → 完了！
```

## 3. 基本の使い方

### タスクを追加する

6つのカテゴリに直接追加できます。カテゴリを省略すると `inbox` に入ります。

#### `inbox` — とりあえず記録したいとき

`/todo inbox` *ブログのアイデアをまとめる*

> 「仕分けるより先に記録」が鉄則。あとで `/todo weekly-review` で整理します。

#### `next` — 次に手をつける作業

`/todo next` *設計書のレビューを終わらせる @PC* `--due` *4/10*

> 「これが次の作業だ」と決めたものだけを入れます。詰め込みすぎに注意。

#### `waiting` — 誰かの返事・完了待ち

`/todo waiting` *田中さんからの見積もり回答 @田中さん*

> 自分ではなく、他の人や外部の何かを待っているときに使います。

#### `someday` — いつかやるかも

- `/todo someday` *英語の勉強を再開する*
- `/todo someday` *新しいツールを試してみる*

> 「今は優先度が低いけど、忘れたくない」ものに使います。

#### `project` — 複数ステップが必要な案件

`/todo project` *新サービスのリリース準備*

> ゴールであって、作業ではありません。具体的な作業は `next` に入れます。

#### `reference` — 後で見返したい情報

- `/todo reference` *会議の議事録（4月5日）*
- `/todo reference` *APIの仕様書URL*

> アクションは不要だけど、残しておきたい情報を保存します。

---

### オプションを使う

タスクに詳細情報を追加できます。

| オプション | 意味 | 例 |
|-----------|------|----|
| `--due` *日付* | 締め切りを設定（形式は下表参照） | `/todo next` *報告書を書く* `--due` *4/10* |
| `--desc` *"テキスト"* | 詳しいメモを追加 | `--desc` *"3章まで読んでから記載"* |
| `--recur` *パターン* | 完了後に自動で再作成（下表参照） | `--recur weekly`（毎週） |
| `--project` *番号* | Issue #番号 のサブタスクとして登録 | `--project` *10* |
| `--priority` *p1\|p2\|p3* | 優先度を設定（省略時はデフォルト `p3`） | `--priority p1`（最優先） |
| `--estimate` *時間* | 見積時間を設定（例：`2h`、`1h30m`、`30m`） | `--estimate 2h` |

**優先度の目安:**

| 値 | 意味 | 表示 |
|----|------|------|
| `p1` | 高優先度（今すぐ対応が必要） | 🔴 |
| `p2` | 中優先度（今週中に対応） | 🟡 |
| `p3` | 低優先度（通常タスク）**[デフォルト]** | （なし） |

#### `--due` に使える形式

| 形式 | 例 | 変換結果 |
|------|----|--------|
| `M/D` | *4/10* | 今年の4月10日 |
| `YYYY-MM-DD` | *2026-04-10* | そのままの日付 |
| `今日` | — | 今日の日付 |
| `明日` | — | 翌日 |
| `明後日` | — | 翌々日 |
| `来週` | — | 7日後 |
| `来月` | — | 翌月同日 |
| `今週末` | — | 次の土曜日 |
| `今月末` | — | 今月の最終日 |
| `来月末` | — | 来月の最終日 |
| `N日後` | *3日後* | N日後 |
| `N週間後` / `N週後` | *2週間後* | N週後 |
| `Nヶ月後` / `Nか月後` | *3ヶ月後* | Nヶ月後 |
| `来週X曜` | *来週月曜* / *来週金曜日* | 来週の指定曜日（月〜日の7パターン。「曜日」でも「曜」でも可） |

#### 繰り返し（`--recur`）の設定値

| 値 | 意味 |
|----|------|
| `daily` | 毎日（完了したら翌日に自動作成） |
| `weekly` | 毎週同じ曜日 |
| `monthly` | 毎月同じ日 |
| `weekdays` | 平日のみ（土日はスキップ） |

毎週月曜に提出するレポート：
`/todo next` *週次進捗レポートを書く @PC* `--due` *4/7* `--recur weekly`

毎日のタスク：
`/todo next` *日報を書く* `--recur daily`

---

### タスクを確認する

- `/todo list` — 全カテゴリを表示（末尾にサマリー: カテゴリ別件数・期限超過・今週期限）※サマリーはフィルタなしのときのみ表示
- `/todo list next` — next だけ表示
- `/todo list` *@PC* — *@PC* コンテキストでフィルタ（全カテゴリ横断）
- `/todo list next` *@PC* — next かつ *@PC* の複合フィルタ（AND条件）
- `/todo list project` *10* — Issue *#10* のサブタスクを表示

### タスクを完了する

- `/todo done` *8* — タスクを完了してクローズ（繰り返し設定があれば次を自動作成）
- `/todo done` *8* `--actual` *1h30m* — 実績時間を記録して完了
- `/todo close` *8* — タスクをクローズ（繰り返し設定があれば次を自動作成）

> `done` と `close` はどちらも同じ動作です。`recur:` が設定されている場合、両方とも次の周期のタスクを自動作成します。`--actual` は完了時にかかった実時間を記録するオプションです（例：`1h`、`30m`、`2h15m`）。

## 4. 日常的な使い方ガイド

### 朝のルーティン（所要時間：5分）

朝に `/todo daily-review`（または `/todo daily-review morning`）を実行するだけで、その日の準備が整います。

`/todo daily-review`

Morning モードでは以下を自動的に進めます：
1. ダッシュボードで全体状況を確認
2. Inbox に未処理がある場合は仕分けを提案
3. 今日やるタスクに期日を設定（番号を指定するだけで `今日` が自動セット）
4. 今日の計画サマリーを表示

---

### 夕方のルーティン（所要時間：5分）

夕方または終業前に `/todo daily-review evening`（または `/todo daily-review pm`）を実行します。

`/todo daily-review evening`

Evening モードでは以下を自動的に進めます：
1. 今日の完了実績を表示
2. 今日期限の未完了タスクを1件ずつ確認（tomorrow / done / someday / skip）
3. 明日期限のタスクを表示して準備を促す
4. 一日のサマリーを表示

> 引数なしで `/todo daily-review` を実行した場合は、時刻で自動判定します（15時前 → Morning、15時以降 → Evening）。

---

### 仕事中の使い方

**思いついたことをすぐ記録する（これが最重要）**

- `/todo inbox` *〇〇の件でフォローアップが必要*
- `/todo inbox` *新しい機能のアイデア*

考えたことをすぐに記録することで、頭の中に「忘れないようにしなければ」という思考を持ち続ける必要がなくなります。

**作業が完了したら**

`/todo done` *8*

**誰かに依頼して待つ状態になったら**

- `/todo move` *5* `waiting`
- `/todo tag` *5 @田中さん* — コンテキストを追加

**場所や状況でフィルタする**

- `/todo list` *@外出中* — 外出中にできることだけ表示
- `/todo list` *@PC* — PCがあるときにできることだけ表示

**期日や説明を後から更新する**

- `/todo due` *8 4/15* — 期日を4月15日に変更
- `/todo desc` *8 詳しい説明を追記* — 説明を更新

---

### 週次レビュー（所要時間：15〜20分）

週次レビューは GTD の核心です。週に一度、「引き出しを整理する時間」です。

> **ポイント：** 週次レビューは「作業する時間」ではなく「整理する時間」です。1つもタスクを実行しなくて OK です。

`/todo weekly-review`

まず**冒頭サマリー**が自動表示されます：
- カテゴリ別タスク件数
- 期限超過タスク（番号・タイトル・期日つき）
- 今週期限タスク（番号・タイトル・期日つき）
- Inbox の未処理件数

**流れ（6ステップ）：**

```
Step 1：inbox を空にする
  → 「これは next か、waiting か、someday か、捨てるか」を一つずつ決める
  → 回答は next / waiting / someday / project / reference / close / skip の7択
  → それ以外を入力すると再質問される
  → 決断が難しいものは someday に入れておく
  → inbox が 0件の場合は自動スキップ

Step 2：next を見直す
  → 「まだ有効か？」「完了済みのものはないか？」
  → 不要になったものはクローズする
  → 0件の場合は自動スキップ

Step 3：waiting を確認する
  → 「催促が必要なものはないか？」
  → 「実は完了しているのでは？」
  → 0件の場合は自動スキップ

Step 4：project を確認する
  → 各プロジェクトに next のタスクが存在するか確認
  → next がないプロジェクトは「次の作業」を決める

Step 5：someday を確認する
  → 「今週始められるものはないか？」
  → 「もう興味がなくなったものはないか？」
  → 0件の場合は自動スキップ

Step 6：完了
  → 処理結果サマリー（Inbox仕分け: N件 など）を表示
  → 整理された next リストを確認して来週に備える
```

週次レビューは金曜の終業前か、月曜の始業前がおすすめです。

## 5. コマンドリファレンス

### タスクを追加する

| コマンド | 説明 |
|----------|------|
| `/todo` *タイトル* | inbox に追加 |
| `/todo inbox` *タイトル* | inbox に追加 |
| `/todo next` *タイトル* | Next Actions に追加 |
| `/todo waiting` *タイトル* | Waiting For に追加 |
| `/todo someday` *タイトル* | Someday/Maybe に追加 |
| `/todo project` *タイトル* | Projects に追加 |
| `/todo reference` *タイトル* | Reference に追加 |

オプション：`--due` *日付*、`--desc` *"説明"*、`--recur` `パターン`、`--project` *番号*、`--priority` *p1|p2|p3*、`--estimate` *時間*

### タスクを表示する

| コマンド | 説明 |
|----------|------|
| `/todo list` | 全カテゴリを表示（末尾にサマリー付き ※フィルタなし時のみ） |
| `/todo list next` | next のみ表示 |
| `/todo list waiting` | waiting のみ表示 |
| `/todo list someday` | someday のみ表示 |
| `/todo list` *@コンテキスト* | コンテキストでフィルタ（全カテゴリ横断） |
| `/todo list next` *@PC* | GTDカテゴリ＋コンテキストの複合フィルタ（AND条件） |
| `/todo list p1` | 優先度 p1 でフィルタ（全カテゴリ横断） |
| `/todo list next p1` | GTDカテゴリ＋優先度の複合フィルタ（AND条件） |
| `/todo list project` *番号* | プロジェクトのサブタスクを表示 |

表示順は各カテゴリ内で **p1 → p2 → p3 → 優先度なし** の順。同優先度内は due 昇順。
p1 の行頭に 🔴、p2 の行頭に 🟡 が付きます。

### ステータスを変更する

| コマンド | 説明 |
|----------|------|
| `/todo move` *番号* `next` | next に移動 |
| `/todo move` *番号* `waiting` | waiting に移動 |
| `/todo move` *番号* `someday` | someday に移動 |
| `/todo move` *番号* `inbox` | inbox に戻す |
| `/todo done` *番号* | 完了してクローズ（繰り返しがあれば次を自動作成） |
| `/todo done` *番号* `--actual` *時間* | 実績時間を記録して完了（例：`--actual 1h30m`） |
| `/todo close` *番号* | クローズ（繰り返しがあれば次を自動作成） |

### 優先度を変更する

| コマンド | 説明 |
|----------|------|
| `/todo priority` *番号* `p1` | 優先度を p1（高）に設定 |
| `/todo priority` *番号* `p2` | 優先度を p2（中）に設定 |
| `/todo priority` *番号* `p3` | 優先度を p3（低）に設定 |
| `/todo priority` *番号* `clear` | 優先度をクリア |

### タスクを更新する

| コマンド | 説明 |
|----------|------|
| `/todo due` *番号 日付* | 期日を更新 |
| `/todo desc` *番号 テキスト* | 説明を更新 |
| `/todo tag` *番号 @ctx1 @ctx2* | コンテキストを追加 |
| `/todo tag rename` *@旧名 @新名* | コンテキスト名を全タスク横断でリネーム（`@` なしでも可） |
| `/todo untag` *番号 @ctx1 @ctx2* | コンテキストを削除 |
| `/todo recur` *番号 daily\|weekly\|monthly\|weekdays* | 繰り返し設定を変更 |
| `/todo recur` *番号* `clear` | 繰り返し設定を解除 |
| `/todo rename` *番号 新タイトル* | タイトルを変更 |
| `/todo link` *アクション番号 プロジェクト番号* | プロジェクトに紐づける |
| `/todo edit` *番号* `--due` *日付* `--priority` *p1* `--desc` *テキスト* | 複数フィールドを同時更新（指定したもののみ変更） |
| `/todo edit` *番号* `--estimate` *時間* | 見積時間を更新（他オプションと組み合わせ可） |

### 検索・統計

| コマンド | 説明 |
|----------|------|
| `/todo search` *キーワード* | オープンIssueをタイトル・本文からキーワード検索 |
| `/todo stats` | カテゴリ別・優先度別件数、期限超過、今週期限、直近7日間の完了数を表示 |

### 一括操作（bulk）

| コマンド | 説明 |
|----------|------|
| `/todo bulk done` *番号 番号 ...* | 複数タスクを一括完了（繰り返しがあれば各自動作成） |
| `/todo bulk move` *番号 番号 ...* `next` | 複数タスクを一括でステータス変更 |
| `/todo bulk tag` *番号 番号 ...* *@ctx1 @ctx2* | 複数タスクにコンテキストを一括追加 |
| `/todo bulk untag` *番号 番号 ...* *@ctx1 @ctx2* | 複数タスクからコンテキストを一括削除 |
| `/todo bulk priority` *番号 番号 ...* `p1` | 複数タスクの優先度を一括変更 |

> 個別の Issue でエラーが発生しても残りは処理を続行します。最後にサマリー（成功件数・失敗件数）が表示されます。

### レビュー・分析

| コマンド | 説明 |
|----------|------|
| `/todo today` | 今日のタスクを表示（期限超過＋今日期限のタスク一覧） |
| `/todo dashboard` | ダッシュボード（全カテゴリのサマリーを俯瞰表示） |
| `/todo report weekly` | 直近1週間の生産性レポート（完了数・所要時間など） |
| `/todo report monthly` | 直近1ヶ月の生産性レポート |
| `/todo report` *Nd* | 直近 N 日間のレポート（例：`/todo report 3d` → 直近3日） |
| `/todo review` | inbox を一件ずつ仕分ける |
| `/todo daily-review` | デイリーレビューを開始（時刻で Morning/Evening を自動判定） |
| `/todo daily-review morning` | Morning モードで開始（ダッシュボード確認→当日計画） |
| `/todo daily-review evening` | Evening モードで開始（完了実績確認→翌日準備） |
| `/todo weekly-review` | 週次レビューを開始（6ステップ） |

### ビュー

よく使うフィルタ条件を「ビュー」として保存しておくと、次回から1コマンドで呼び出せます。

| コマンド | 説明 |
|----------|------|
| `/todo view list` | 保存済みビューの一覧を表示 |
| `/todo view save` *名前* `next` *@PC* | フィルタ条件をビューとして保存（GTDカテゴリ・コンテキスト・優先度を組み合わせ可） |
| `/todo view use` *名前* | 保存したビューでタスクを表示 |
| `/todo view` *名前* | `view use` の省略形 |
| `/todo view delete` *名前* | ビューを削除 |

### アーカイブ（完了済みタスク）

## 6. 応用機能

### プロジェクト管理

複数の作業が必要な案件は `project` として管理し、個々の作業を `--project` で紐づけます。

プロジェクトを登録する（例：Issue #12 が作成される）：
`/todo project` *社内システム移行*

サブタスクをプロジェクトに紐づけて追加する：
- `/todo next` *現行システムの機能一覧を作成 @PC* `--project` *12* `--due` *4/10*
- `/todo next` *ステークホルダーにヒアリングする @会社* `--project` *12* `--due` *4/15*
- `/todo someday` *移行後のトレーニング計画を立てる* `--project` *12*

プロジェクトのタスク一覧を確認する：
`/todo list project` *12*

既存タスクをプロジェクトに紐づける：
`/todo link` *8 12*

**週次レビューのときに確認すること：** 各プロジェクトに `next` のタスクが必ず1つ以上あるかを確認します。next がないプロジェクトは「止まっているプロジェクト」です。

---

### テンプレート

毎週・毎月同じ設定でタスクを作る場合に便利です。

テンプレートを保存する：
`/todo template save` *週次レポート* `next` *@PC* `--recur weekly`

テンプレートの詳細を確認する：
`/todo template show` *週次レポート*

テンプレートからタスクを作成する（タイトルを指定）：
`/todo template use` *週次レポート 今週の進捗レポートを書く*

テンプレートからタスクを作成する（タイトル省略 → テンプレート名がそのままタイトルになる）：
`/todo template use` *週次レポート*

既存のタスク（例：Issue #15）からテンプレートを作成する：
`/todo template save` *定例作業* `from` *15*

テンプレート一覧を確認する：
`/todo template list`

**`--due-offset` について（保存時に指定）：**  
`--due-offset` *3* を指定すると、テンプレートを使った日から3日後が自動的に期日になります。「来週の何日」と毎回計算する手間がなくなります。  
`--due` と同時に指定した場合は `--due-offset` が優先されます。

**`template save` で指定できるオプション一覧：**

| オプション | 説明 | 例 |
|-----------|------|-----|
| GTDキーワード | タスクのカテゴリ（省略時 inbox） | `next` |
| *@コンテキスト* | コンテキストラベル（複数可） | *@PC @会社* |
| `--priority` *p1\|p2\|p3* | 優先度（省略時 p3） | `--priority p1` |
| `--due-offset` *N* | 作成日から N 日後を期日に設定 | `--due-offset 7` |
| `--due` *日付* | 固定の期日（due-offset と同時指定時は due-offset 優先） | `--due 4/30` |
| `--recur` *パターン* | 繰り返しパターン | `--recur weekly` |
| `--project` *番号* | プロジェクトへの紐づけ | `--project 10` |
| `--desc` *"説明"* | 説明文 | `--desc "詳細は〜"` |

**対話形式での作成：**  
`/todo template save` *テンプレート名* のように名前だけ指定すると、対話形式で以下の項目を順番に入力できます：

1. GTDラベル（next / waiting / someday / inbox / project / reference、省略すると inbox）
2. コンテキスト（例：*@PC @会社*、スペース区切りで複数入力可）
3. 優先度（p1 / p2 / p3、省略すると p3）
4. 期日オフセット（作成日から何日後か、例：*7*）
5. 繰り返しパターン（daily / weekly / monthly / weekdays）
6. プロジェクト番号（数字のみ）
7. 説明文

## 7. よくある質問・シナリオ集

### よくある質問

**Q1. `inbox` と `next` の違いがわかりません**

> `inbox` は「まだ考えていないもの」、`next` は「次にやると決めたもの」です。  
> 「とにかく記録したい」ときは `inbox`、「これが次の作業だ」と決断したら `/todo move` *番号* `next` で移します。  
> 迷ったら `inbox` に入れましょう。週次レビューのときに整理します。

**Q2. `project` の中のタスクはどう管理しますか？**

> プロジェクトに紐づくタスクは `--project` *番号* オプションで作成します。

- `/todo project` *新サービスのリリース* — Issue *#10* が作成される
- `/todo next` *要件定義書を書く* `--project` *10*
- `/todo next` *デザインレビューをする* `--project` *10*
- `/todo list project` *10* — サブタスク一覧を確認

**Q3. 繰り返しタスクが重複して増えてしまいました**

> `/todo done` *番号* と `/todo close` *番号* はどちらも同じ動作です。`--recur` が設定されているタスクを完了にすると、**必ず次の周期のタスクが自動作成されます**。  
> 不要な繰り返しタスクを止めたい場合は、`/todo recur` *番号* `clear` で繰り返し設定を解除してから `/todo done` でクローズしてください。  
> または、単純に残ってしまった重複タスクをそのまま `/todo done` で完了にするだけでも構いません。

**Q4. コンテキストを後から追加・変更したい**

> 追加は `/todo tag` *番号 @新しいコンテキスト* でできます。  
> 削除したい場合は `/todo untag` *番号 @コンテキスト* を使います。  
> 全タスクのコンテキスト名を変更したい場合は `/todo label rename` *@旧名 @新名* を使います。

**Q5. タイトルを変更したい**

> `/todo rename` *番号 新タイトル* で変更できます。日本語やスペースを含むタイトルも安全に扱えます。

**Q6. 週次レビューはどのくらいの頻度でやればいいですか？**

> 名前の通り、週に1回が推奨です。金曜の終業前か、月曜の始業前がおすすめです。  
> 最初は毎週できなくても大丈夫です。2〜3週間に1回でも、やらないよりずっと効果があります。

**Q7. `done` と `close` の違いは何ですか？**

> 実質的に同じ動作です。どちらも Issue をクローズし、`--recur` が設定されている場合は次の周期のタスクを自動作成します。コマンド名が異なるだけで処理内容は変わりません。

### シナリオ集

#### シナリオA：新しいプロジェクトを受け持ったとき

1\. プロジェクトを登録する：
`/todo project` *社内システム移行*
（→ Issue *#12* が作成される）

2\. 最初のアクションを決めて紐づける：
`/todo next` *現行システムの機能一覧を作成* `--project` *12* `@PC` `--due` *4/10*

3\. 後でやることは someday に入れておく：
`/todo someday` *移行後のトレーニング計画を立てる* `--project` *12*

4\. プロジェクト全体を確認する：
`/todo list project` *12*

#### シナリオB：外出先で確認・作業するとき

外出中にできるタスクだけ確認する：
`/todo list` *@外出中*

完了したタスクを記録する：
`/todo done` *7*

外出中に思いついたことを記録する（帰宅後に仕分ければよい）：
`/todo inbox` *〇〇さんへのフォローアップをする*

#### シナリオC：週次レビューで inbox が10件溜まっているとき

`/todo weekly-review` を実行すると、対話形式で順番に仕分けが進みます。

```
# Step 1 で inbox の各タスクを仕分ける
# 例：「#3 〇〇の件を確認する」→ next / waiting / someday / project / reference / close / skip ?
# → next と答える → 次のタスクへ
# → （繰り返す）

# Step 2 以降は next / waiting / project / someday の順に確認が進む
```

> **コツ：** 週次レビューの最初は時間がかかりますが、続けるうちに15分以内に終わるようになります。inbox に溜め込まずに少しずつ仕分けるのがポイントです。
